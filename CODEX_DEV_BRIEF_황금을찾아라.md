# CODEX 개발 브리프 — 황금을 찾아라

## 0. 프로젝트 개요

**서비스명:** 황금을 찾아라  
**서비스 한 줄 설명:** 서울시 공공데이터 기반으로, MZ세대 대상 카페 창업자가 원하는 운영 시간대에 실제 구매 전환이 높은 행정동을 추천하는 상권 추천 서비스.

본 서비스는 단순히 유동인구가 많은 지역을 보여주는 것이 아니라, **유동인구 대비 실제 카페 구매가 발생하는 지역**을 추천하는 것을 목표로 한다.

MVP에서는 다음 범위만 구현한다.

- 업종: 커피·음료
- 타겟: 2030세대
- 추천 단위: 서울시 행정동
- 주요 입력: 희망 운영 시간대
- 주요 출력: 추천 행정동 순위, 추천점수, 핵심 지표, 추천 사유, 비교 정보

---

## 1. 개발 목표

백엔드와 프론트엔드를 구현한다.

### 백엔드 목표

- `cafe_area_features.csv`를 읽어 추천 API 제공
- 사용자가 선택한 시간대에 따라 행정동별 추천점수 계산
- 추천 행정동 Top N 반환
- 추천 이유 생성
- 두 행정동 비교 API 제공

### 프론트엔드 목표

- 사용자가 희망 운영 시간대를 선택할 수 있는 화면 구현
- 추천 결과 카드 목록 구현
- 추천 이유와 핵심 지표 표시
- 두 행정동 비교 화면 구현
- 서비스 컨셉과 차별점을 알기 쉽게 보여주는 기본 UI 구현

## 2. 핵심 사용자 요구사항

### 2.1 사용자는 희망 운영 시간대를 선택할 수 있어야 한다

MVP에서 업종과 타겟은 고정한다.

```text
업종: 커피·음료
타겟: 2030세대
```

사용자가 선택하는 입력값은 다음이다.

```text
희망 운영 시간대:
- 새벽
- 오전
- 점심
- 오후
- 저녁
- 심야
```

시간대 매핑은 다음과 같다.

```text
새벽: 00~06
오전: 06~11
점심: 11~14
오후: 14~17
저녁: 17~21
심야: 21~24
```

### 2.2 사용자는 추천 행정동 순위를 볼 수 있어야 한다

추천 결과에는 다음 정보가 포함되어야 한다.

```text
- 순위
- 행정동 코드
- 행정동명
- 추천점수
- 2030 매출비율
- 2030 유동인구비율
- 카페전환효율
- 선택시간대 매출비중
- 객단가
- 추천 사유
```

### 2.3 사용자는 추천 이유를 이해할 수 있어야 한다

추천 결과는 단순 점수만 제공하지 말고, 데이터 기반으로 왜 추천되었는지 설명해야 한다.

예시:

```text
○○동은 2030세대 매출비율이 높고, 사용자가 선택한 저녁 시간대의 카페 매출비중이 높은 지역입니다.
또한 유동인구 대비 카페 구매 전환효율이 높아 실제 카페 소비가 활발한 상권으로 판단됩니다.
```

MVP에서는 생성형 AI API 연동이 없어도 된다. 우선은 **정량 지표 기반 rule-based 설명 생성 함수**를 구현한다.  
추후 생성형 AI 연동 시 `reason.service.js`만 교체할 수 있도록 모듈화한다.

### 2.4 사용자는 후보 상권을 비교할 수 있어야 한다

추천 결과 중 두 행정동을 선택하여 다음 지표를 비교할 수 있어야 한다.

```text
- 2030 매출비율
- 2030 유동인구비율
- 총 유동인구
- 카페전환효율
- 선택시간대 매출비중
- 시간대별 매출비중
- 객단가
```

### 2.5 사용자는 운영 전략 가이드를 받을 수 있어야 한다

선택 시간대와 추천 지역의 특성을 기반으로 간단한 운영 전략 메시지를 제공한다.

예시:

```text
오전형 상권: 출근길 테이크아웃, 빠른 회전율 중심 전략
오후형 상권: 카공, 디저트, 브런치 메뉴 강화 전략
저녁형 상권: 데이트, 모임, 디저트 중심 카페 전략
```

## 4. 추천점수 로직

### 4.1 추천점수 산식

MVP 추천점수는 다음 산식을 사용한다.

```text
MZ카페 추천점수 =
    카페전환효율 점수 × 0.35
  + 2030 매출비율 점수 × 0.30
  + 선택시간대 매출비중 점수 × 0.25
  + 객단가 점수 × 0.10
```

각 지표는 Min-Max Scaling으로 0~1 범위로 정규화한다.

```text
정규화점수 = (현재값 - 최솟값) / (최댓값 - 최솟값)
```

최종 표시 점수는 0~100점으로 변환한다.

```text
표시점수 = 추천점수 × 100
```

소수점 1자리까지 표시한다.

### 4.2 선택시간대 매출비중

사용자가 `저녁`을 선택하면 다음 컬럼을 사용한다.

```text
저녁_매출비중
```

fallback으로 다음 컬럼을 사용할 수 있다.

```text
17~21_매출비중
```

사용자가 `오전`을 선택하면 다음 컬럼을 사용한다.

```text
오전_매출비중
```

fallback:

```text
06~11_매출비중
```

### 4.3 시간대 매핑 객체

백엔드와 프론트엔드 모두 동일한 시간대 매핑을 사용한다.

```js
const TIME_OPTIONS = [
  { label: "새벽", range: "00~06", description: "새벽 시간대" },
  { label: "오전", range: "06~11", description: "출근길·오전 커피 수요" },
  { label: "점심", range: "11~14", description: "점심 식사 후 카페 소비" },
  { label: "오후", range: "14~17", description: "카공·디저트·브런치 수요" },
  { label: "저녁", range: "17~21", description: "데이트·모임·여가 소비" },
  { label: "심야", range: "21~24", description: "늦은 시간대 소비" },
];
```

## 5. 추천 이유 생성 로직

MVP에서는 LLM API 없이 rule-based로 설명한다.

### 5.1 추천 이유 구성 요소

추천 이유는 다음 지표를 기반으로 생성한다.

```text
- 2030 매출비율
- 카페전환효율
- 선택시간대 매출비중
- 객단가
- 2030 유동인구비율
```

### 5.2 예시 규칙

```text
카페전환효율이 상위 25%이면:
→ "유동인구 대비 카페 구매 전환효율이 높아 실제 카페 소비가 활발한 지역입니다."

2030 매출비율이 평균 이상이면:
→ "2030세대 매출비율이 높아 MZ 타겟 카페에 적합합니다."

선택시간대 매출비중이 평균 이상이면:
→ "사용자가 선택한 시간대의 카페 매출비중이 높아 해당 시간대 운영에 유리합니다."

객단가가 평균 이상이면:
→ "평균 객단가가 비교적 높아 디저트·프리미엄 메뉴 전략을 고려할 수 있습니다."
```

최소 2개, 최대 4개의 추천 이유를 반환한다.

### 6.5 백엔드 구현 주의사항

- CSV는 서버 시작 시 한 번 로드하고 메모리에 저장한다.
- 숫자 컬럼은 반드시 number로 변환한다.
- `NaN`, `Infinity`, `undefined`는 JSON 응답에 포함하지 않는다. 값이 없으면 `null` 또는 `0`으로 처리한다.
- 잘못된 시간대 입력은 400 에러를 반환한다.
- 추천점수는 API 호출 시 선택한 시간대 기준으로 동적으로 계산한다.
- 하드코딩된 가짜 추천 결과를 만들지 않는다.
- 추천 결과는 반드시 CSV 데이터 기반으로 계산한다.

## AI / Local LLM Integration

본 서비스는 추천점수 계산과 자연어 설명 생성을 분리한다.

### 역할 분리

- 추천점수 계산은 백엔드의 알고리즘이 담당한다.
- Local LLM은 추천 결과를 사용자에게 이해하기 쉬운 자연어로 설명하는 역할만 수행한다.
- LLM은 추천 순위를 직접 결정하거나 새로운 수치를 생성하면 안 된다.

### AI 설명 기능 요구사항

Backend에 `aiReason.service.js`를 구현한다.

입력 데이터:
- 행정동명
- 선택 시간대
- 추천점수
- 2030 매출비율
- 카페전환효율
- 선택시간대 매출비중
- 객단가
- 피크타임

출력 데이터:
- 추천 사유 요약
- 시간대 적합성 설명
- 운영 전략 제안

### Local LLM 방식

개발 환경에서는 로컬 LLM을 사용한다.

예상 구조:

- Ollama 또는 llama.cpp 기반 로컬 LLM 서버 실행
- Backend에서 HTTP 요청으로 LLM 호출
- LLM 사용이 실패하면 rule-based template 설명으로 fallback 처리

### API 예시

GET /api/recommend?time=저녁&limit=5&ai=true

응답 예시:
