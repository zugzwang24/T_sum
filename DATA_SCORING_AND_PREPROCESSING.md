# 카페 상권 데이터 전처리와 추천 점수 보정 설명

이 문서는 `황금을 찾아라` 서비스에서 카페 상권 추천 데이터를 어떻게 전처리했고, 추천 점수, 이상치 보정, 데이터 신뢰도, 신뢰도 보정계수를 어떤 의도로 만들었는지 정리한 문서다. 보고서나 발표 자료를 작성할 때 GPT가 읽고 이해할 수 있도록 구현 기준과 해석 기준을 함께 설명한다.

## 1. 전체 구조 요약

서비스의 데이터 흐름은 다음 순서로 구성된다.

```text
raw CSV
  -> pop.py
  -> sales.py
  -> process.py
  -> cafe_area_features.csv
  -> backend/src/dataStore.js에서 조건별 추천 점수 계산
  -> frontend에서 TOP 10, 상세 지표, 신뢰도, 주의사항 표시
```

중요한 설계 방향은 `CSV에 최종 추천 점수를 고정 저장하지 않는다`는 점이다. CSV에는 행정동별 feature만 저장하고, 사용자가 선택한 시간대, 타깃 연령대, 이상치 보정 여부에 따라 백엔드에서 추천 점수를 다시 계산한다.

## 2. 사용한 원천 데이터

현재 전처리에 사용하는 원천 데이터는 다음 두 가지다.

```text
data_analysis/data/raw/유동인구.csv
data_analysis/data/raw/추정매출.csv
```

두 파일 모두 서울 공공데이터 기반이며, 원본 인코딩은 `cp949`로 읽는다.

현재 전처리 기준 기간은 2025년 1~4분기다.

```python
TARGET_QUARTERS = [20251, 20252, 20253, 20254]
```

업종은 기본적으로 `커피-음료`만 필터링한다.

## 3. 유동인구 전처리

유동인구 전처리는 다음 파일에서 수행한다.

```text
data_analysis/data/processed/pop.py
```

### 3.1 처리 방식

1. `유동인구.csv`를 읽는다.
2. 2025년 1~4분기 데이터만 남긴다.
3. 행정동 단위로 그룹화한다.
4. 분기별 유동인구는 평균값으로 집계한다.

유동인구는 매출처럼 누적 합계로 보기보다, 특정 기간의 유동 규모를 나타내는 지표이므로 행정동별 평균을 사용했다.

### 3.2 생성되는 주요 지표

```text
총_유동인구_수
연령대_10_유동인구_수
연령대_20_유동인구_수
연령대_30_유동인구_수
연령대_40_유동인구_수
연령대_50_유동인구_수
연령대_60_이상_유동인구_수
시간대_00_06_유동인구_수
시간대_06_11_유동인구_수
시간대_11_14_유동인구_수
시간대_14_17_유동인구_수
시간대_17_21_유동인구_수
시간대_21_24_유동인구_수
```

기존 분석과의 연결을 위해 2030 유동인구도 별도 feature로 유지했다.

```text
2030_유동인구 = 연령대_20_유동인구_수 + 연령대_30_유동인구_수
2030_유동인구비율 = 2030_유동인구 / 총_유동인구_수
```

시간대별 유동인구비중도 계산한다.

```text
새벽_유동인구비중 = 시간대_00_06_유동인구_수 / 총_유동인구_수
오전_유동인구비중 = 시간대_06_11_유동인구_수 / 총_유동인구_수
점심_유동인구비중 = 시간대_11_14_유동인구_수 / 총_유동인구_수
오후_유동인구비중 = 시간대_14_17_유동인구_수 / 총_유동인구_수
저녁_유동인구비중 = 시간대_17_21_유동인구_수 / 총_유동인구_수
심야_유동인구비중 = 시간대_21_24_유동인구_수 / 총_유동인구_수
```

## 4. 매출 데이터 전처리

매출 전처리는 다음 파일에서 수행한다.

```text
data_analysis/data/processed/sales.py
```

### 4.1 처리 방식

1. `추정매출.csv`를 읽는다.
2. 2025년 1~4분기만 남긴다.
3. 업종명이 `커피-음료`인 행만 남긴다.
4. 행정동 단위로 그룹화한다.
5. 매출금액, 매출건수, 연령대별 매출건수, 시간대별 매출금액은 행정동별로 합산한 뒤 `집계_기간수`로 나누어 평균화한다.

원천 컬럼명은 `당월_매출_금액`, `당월_매출_건수`이므로 월 단위 지표로 해석한다. 다만 전처리에서는 2025년 1~4분기를 함께 불러오기 때문에, 최종 추천 데이터에서는 선택 기간 합산값이 아니라 기간 평균값을 사용한다.

### 4.2 생성되는 주요 지표

```text
당월_매출_금액
당월_매출_건수
총매출건수
집계_기간수
연령대_10_매출_건수
연령대_20_매출_건수
연령대_30_매출_건수
연령대_40_매출_건수
연령대_50_매출_건수
연령대_60_이상_매출_건수
연령대별 매출금액
시간대별 매출금액
```

평균화 방식은 다음과 같다.

```text
집계_기간수 = 행정동별 기준_년분기_코드 고유 개수
당월_매출_금액 = 선택 기간 합산 당월_매출_금액 / 집계_기간수
당월_매출_건수 = 선택 기간 합산 당월_매출_건수 / 집계_기간수
총매출건수 = 선택 기간 합산 총매출건수 / 집계_기간수
2030_매출건수 = 선택 기간 합산 2030_매출건수 / 집계_기간수
```

연령대별 매출건수, 연령대별 매출금액, 시간대별 매출금액도 동일하게 `집계_기간수`로 나누어 평균화한다.

객단가는 다음과 같이 계산한다.

```text
객단가 = 당월_매출_금액 / 당월_매출_건수
```

2030 매출비율은 기존 분석 흐름을 유지하기 위해 별도로 계산한다.

```text
2030_매출건수 = 연령대_20_매출_건수 + 연령대_30_매출_건수
2030_매출비율 = 2030_매출건수 / 총매출건수
```

시간대별 매출비중은 다음과 같이 계산한다.

```text
새벽_매출비중 = 시간대_00~06_매출_금액 / 당월_매출_금액
오전_매출비중 = 시간대_06~11_매출_금액 / 당월_매출_금액
점심_매출비중 = 시간대_11~14_매출_금액 / 당월_매출_금액
오후_매출비중 = 시간대_14~17_매출_금액 / 당월_매출_금액
저녁_매출비중 = 시간대_17~21_매출_금액 / 당월_매출_금액
심야_매출비중 = 시간대_21~24_매출_금액 / 당월_매출_금액
```

가장 매출이 큰 시간대는 `시간대추천`으로 저장한다.

## 5. 최종 feature 병합

최종 병합은 다음 파일에서 수행한다.

```text
data_analysis/data/processed/process.py
```

### 5.1 병합 방식

매출 feature와 유동인구 feature를 다음 key로 병합한다.

```text
행정동_코드
행정동_코드_명
```

최종 파일은 다음 위치에 저장된다.

```text
data_analysis/data/processed/cafe_area_features.csv
```

현재 최종 파일은 421개 행정동, 59개 컬럼으로 구성된다.

### 5.2 최종 추가 지표

월 유동인구 추정값은 다음과 같이 계산한다.

```text
월_유동인구추정 = 총_유동인구_수 * (365.25 / 12 / 7)
```

전체 카페전환효율은 월평균 매출건수와 월 유동인구 추정치를 맞춰 다음과 같이 계산한다.

```text
카페전환효율 = 당월_매출_건수 / 월_유동인구추정
```

2030 카페전환효율도 월평균 2030 매출건수와 월 2030 유동인구 추정치를 기준으로 계산한다.

```text
2030_카페전환효율 = 2030_매출건수 / (2030_유동인구 * (365.25 / 12 / 7))
```

## 6. 카페전환효율과 타깃 전환효율의 차이

여기서 가장 헷갈릴 수 있는 부분이 있다.

현재 백엔드에서는 타깃 연령대를 선택하면 `targetConversionRate`를 동적으로 계산한다.

```text
targetSalesCount = 선택한 연령대 매출건수 합계
targetPopulation = 선택한 연령대 유동인구 합계
targetConversionRate = targetSalesCount / (targetPopulation * (365.25 / 12 / 7))
```

하지만 현재 추천 점수에 직접 들어가는 전환효율은 `targetConversionRate`가 아니라 CSV의 `카페전환효율`이다.

```text
conversionRate = 카페전환효율
```

즉 현재 구현 기준은 다음과 같다.

```text
추천 점수에 사용되는 전환효율: 전체 카페전환효율
상세 지표에 표시되는 타깃 전환효율: targetConversionRate
```

이렇게 설계한 이유는 타깃 전환효율이 분모인 타깃 유동인구에 크게 흔들릴 수 있기 때문이다. 예를 들어 특정 행정동에서 선택 연령대 유동인구가 작으면 `targetConversionRate`가 과도하게 커질 수 있다. 따라서 추천 점수에는 더 넓은 기준의 전체 카페전환효율을 사용하고, 타깃 적합도는 `타깃 매출비율`, `타깃 유동인구비율`, `targetConversionRate`를 함께 참고하도록 했다.

만약 나중에 정말 타깃 전환효율을 점수에 넣고 싶다면, 백엔드에서 `conversionRate` 대신 `targetConversionRate`를 score field에 넣고, 이상치 보정과 신뢰도 계산도 타깃 기준으로 다시 조정해야 한다.

## 7. 백엔드에서 동적으로 계산하는 타깃 지표

사용자는 API에서 타깃 연령대를 선택할 수 있다.

```text
targetAges=20,30
targetAges=40,50
targetAges=all
```

백엔드는 선택된 연령대의 매출건수와 유동인구를 합산한다.

```text
타깃 매출건수 = 선택 연령대 매출건수 합계
타깃 유동인구 = 선택 연령대 유동인구 합계
타깃 매출비율 = 타깃 매출건수 / 총매출건수
타깃 유동인구비율 = 타깃 유동인구 / 총_유동인구_수
타깃 전환효율 = 타깃 월평균 매출건수 / (타깃 유동인구 * (365.25 / 12 / 7))
```

따라서 같은 행정동이라도 `20,30대`를 타깃으로 볼 때와 `40,50대`를 타깃으로 볼 때 추천 점수와 순위가 달라질 수 있다.

## 8. 추천 점수 계산 방식

추천 점수는 `backend/src/dataStore.js`에서 계산한다.

현재 추천 점수의 구성 요소는 다음 네 가지다.

```text
카페전환효율: 35%
타깃 매출비율: 30%
선택 시간대 매출비중: 25%
객단가: 10%
```

코드 기준 가중치는 다음과 같다.

```js
const SCORE_WEIGHTS = {
  conversionRate: 0.35,
  targetSalesRatio: 0.3,
  selectedTimeSalesRatio: 0.25,
  averagePrice: 0.1,
};
```

각 지표는 바로 더하지 않고 먼저 0~1 범위로 정규화한다.

```text
normalizedValue = (value - min) / (max - min)
```

그리고 0보다 작으면 0, 1보다 크면 1로 제한한다.

```text
normalizedValue = clamp(normalizedValue, 0, 1)
```

기본점수는 다음과 같다.

```text
baseScore =
  normalized(카페전환효율) * 0.35
+ normalized(타깃 매출비율) * 0.30
+ normalized(선택 시간대 매출비중) * 0.25
+ normalized(객단가) * 0.10
```

화면에는 100점 만점으로 표시한다.

```text
baseScoreDisplay = baseScore * 100
```

## 9. 이상치 보정 점수 토글

프론트에는 `이상치 보정 점수 사용` 토글이 있다. 이 토글은 데이터를 삭제하는 기능이 아니다. 이상치를 제거해서 행을 없애는 것이 아니라, 점수 계산에서 정규화 기준을 더 안정적으로 바꾸는 기능이다.

일반 점수에서는 각 지표의 최솟값과 최댓값을 사용한다.

```text
min = 전체 최솟값
max = 전체 최댓값
```

이상치 보정 점수에서는 1% 지점과 99% 지점을 정규화 기준으로 사용한다.

```text
min = 1 percentile
max = 99 percentile
```

코드 기준으로는 다음 지표들에 적용된다.

```text
카페전환효율
타깃 매출비율
선택 시간대 매출비중
객단가
```

이 방식은 winsorizing과 비슷한 효과를 낸다. 실제 값을 바꾸거나 행을 삭제하지는 않지만, 상위 1% 또는 하위 1% 수준의 극단값이 전체 점수 분포를 과하게 흔드는 것을 줄인다.

예를 들어 어떤 행정동의 카페전환효율이 지나치게 높으면 일반 min-max scaling에서는 그 값이 max가 되어 다른 상권의 점수를 눌러버릴 수 있다. 이상치 보정에서는 99 percentile을 max로 보기 때문에 그런 극단값의 영향이 줄어든다.

## 10. 데이터 신뢰도 점수

추천 점수가 높더라도 데이터가 불안정하면 실제 서비스에서는 위험하다. 그래서 별도의 `데이터 신뢰도`를 만들었다.

데이터 신뢰도는 추천 점수와 다른 목적의 지표다.

```text
추천 점수: 이 상권이 조건에 얼마나 잘 맞는가
데이터 신뢰도: 이 추천 결과를 얼마나 안정적으로 해석할 수 있는가
```

### 10.1 신뢰도 구성 요소

데이터 신뢰도는 네 가지 요소로 구성된다.

```text
총 매출건수 규모: 35%
총 유동인구 규모: 25%
타깃 유동인구 규모: 25%
카페전환효율 이상치 여부: 15%
```

코드 기준 산식은 다음과 같다.

```text
dataQualityScore =
  salesVolumeScore * 0.35
+ populationScore * 0.25
+ targetPopulationScore * 0.25
+ outlierScore * 0.15
```

마지막에 100을 곱해서 100점 만점으로 만든다.

```text
dataQualityScoreDisplay = dataQualityScore * 100
```

### 10.2 각 요소의 계산 방식

매출건수 규모 점수:

```text
salesVolumeScore = normalized(총 매출건수, p10, p75)
```

총 유동인구 규모 점수:

```text
populationScore = normalized(총 유동인구, p10, p75)
```

타깃 유동인구 규모 점수:

```text
targetPopulationScore = normalized(타깃 유동인구, p10, p75)
```

여기서 `p10`은 하위 10% 지점, `p75`는 상위 25% 지점이다. 평균이나 최댓값 대신 percentile을 사용한 이유는, 표본 규모가 매우 큰 일부 중심 상권이 신뢰도 기준을 독점하지 않게 하기 위해서다.

이상치 점수:

```text
카페전환효율이 하위 1% 또는 상위 1%이면 outlierScore = 0.35
그 외에는 outlierScore = 1
```

이상치라고 해서 무조건 0점을 주지 않고 0.35를 주는 이유는, 실제로 특수하게 전환효율이 높은 상권일 가능성도 있기 때문이다. 완전히 제거하기보다는 `주의가 필요한 후보`로 남긴다.

### 10.3 신뢰도 등급

신뢰도 점수는 다음 등급으로 표시한다.

```text
80점 이상: 높음
60점 이상: 보통
40점 이상: 주의
40점 미만: 낮음
```

## 11. 신뢰도 보정계수

처음에는 신뢰도를 화면에만 표시했다. 하지만 그렇게 하면 신뢰도 40점짜리 상권도 기본 점수가 높으면 1등으로 올라오는 문제가 있었다. 사용자는 TOP 1을 가장 좋은 추천으로 받아들이기 때문에, 데이터가 불안정한 상권이 상단에 노출되는 것은 서비스 신뢰도를 떨어뜨린다.

그래서 최종 점수에 신뢰도 보정계수를 곱했다.

### 11.1 실제 구현 수식

문서나 설명에서 헷갈리지 않도록 변수명을 분리하면 다음과 같다.

```text
Q = 데이터 신뢰도 점수, 0~100
L = 최저 보정계수, 현재 0.55
```

현재 구현된 신뢰도 보정계수는 다음이다.

```text
reliabilityFactor = L + (Q / 100) * (1 - L)
```

현재 `L = 0.55`이므로 실제 식은 다음과 같다.

```text
reliabilityFactor = 0.55 + (Q / 100) * 0.45
```

최종 점수는 다음과 같다.

```text
finalScore = baseScore * reliabilityFactor
```

100점 만점 표시 기준으로는 다음과 같다.

```text
finalScoreDisplay = baseScore * reliabilityFactor * 100
```

### 11.2 왜 이런 형태인가

이 수식은 선형 보간이다. 신뢰도가 0점일 때도 점수를 완전히 0으로 만들지 않고, 최소 55%는 인정한다. 신뢰도가 100점이면 보정계수가 1이 되어 기본점수를 그대로 사용한다.

```text
Q = 0   -> reliabilityFactor = 0.55
Q = 40  -> reliabilityFactor = 0.55 + 0.40 * 0.45 = 0.73
Q = 60  -> reliabilityFactor = 0.55 + 0.60 * 0.45 = 0.82
Q = 80  -> reliabilityFactor = 0.55 + 0.80 * 0.45 = 0.91
Q = 100 -> reliabilityFactor = 1.00
```

즉 신뢰도가 낮은 상권은 점수가 크게 깎이지만, 완전히 사라지지는 않는다.

### 11.3 왜 0.55인가

0.55는 통계적으로 증명된 상수가 아니라 MVP 단계에서 선택한 정책값이다. 이 값은 다음 의도를 반영한다.

```text
1. 신뢰도가 낮은 상권도 후보군에서 완전히 제거하지 않는다.
2. 하지만 신뢰도 40점 수준의 상권이 1등으로 올라오는 것은 막는다.
3. 기존 0.8 하한은 감점이 너무 약했기 때문에 더 강한 하한이 필요했다.
4. 0.55는 최대 45%까지 점수를 낮출 수 있어, 데이터 불안정성을 실제 순위에 반영한다.
```

초기에는 다음 형태에 가까웠다.

```text
reliabilityFactor = 0.8 + (Q / 100) * 0.2
```

이 경우 신뢰도 40점이라도 보정계수는 0.88이었다.

```text
0.8 + 0.4 * 0.2 = 0.88
```

즉 점수가 12%밖에 깎이지 않아, 전환효율 이상치가 여전히 TOP 1에 올라오는 문제가 있었다.

현재는 다음과 같다.

```text
0.55 + 0.4 * 0.45 = 0.73
```

신뢰도 40점인 상권은 기본점수의 73%만 인정되므로, 불안정한 이상치가 상위 추천을 독점하기 어렵다.

## 12. 신뢰도 60점 이상 우선 정렬

신뢰도 보정계수만으로는 부족할 수 있기 때문에 정렬 정책도 추가했다.

현재 기본 정렬 정책은 다음과 같다.

```text
신뢰도 60점 이상: 안정 추천
신뢰도 60점 미만: 검토 후보
```

API 기준 기본 최소 신뢰도는 60점이다.

```js
const DEFAULT_MIN_RECOMMENDATION_QUALITY = 60;
```

정렬할 때는 먼저 `안정 추천`을 상단에 배치하고, 그 안에서 점수순으로 정렬한다. 신뢰도 60점 미만의 `검토 후보`는 점수가 높더라도 기본 TOP 상단에 올라오지 않는다.

단, 분석용으로 보고 싶을 때는 `allowLowConfidenceTop=true` 옵션을 줄 수 있다. 이 경우 신뢰도와 관계없이 단순 점수순으로 볼 수 있다.

## 13. 주의사항 생성

데이터 신뢰도가 낮거나 이상치 가능성이 있으면 `cautions`에 주의사항을 넣는다.

예시:

```text
매출건수 표본이 작은 편이라 전환효율이 실제보다 크게 흔들릴 수 있습니다.
총 유동인구가 낮은 편이라 특정 매장의 영향이 크게 반영됐을 수 있습니다.
선택한 타깃 연령대 유동인구 표본이 작아 타깃 적합도 해석에 주의가 필요합니다.
카페전환효율이 상위 1% 수준으로 높아 이상치 가능성이 있습니다.
카페전환효율이 하위 1% 수준으로 낮아 데이터 누락 또는 특수 상권 여부를 확인해야 합니다.
선택 시간대 매출 집중도가 유동인구보다 높게 나타나 재방문/목적형 소비 영향일 수 있습니다.
```

이 주의사항은 사용자가 점수를 맹신하지 않고, 왜 해당 상권이 조심스러운 후보인지 이해하도록 돕는다.

## 14. 최종 API 응답에서 확인할 수 있는 값

추천 API 예시:

```text
GET /api/recommend?time=저녁&targetAges=20,30&limit=10&useAdjustedScore=true&minQualityScore=60
```

응답에는 다음 값들이 포함된다.

```text
score: 신뢰도 보정이 반영된 최종 점수
baseScore: 신뢰도 보정 전 기본 점수
reliabilityFactor: 신뢰도 보정계수
dataQuality: 데이터 신뢰도 점수와 등급
cautions: 주의사항
reviewRequired: 검토 후보 여부
recommendationTier: 안정 추천 또는 검토 후보
scoreBreakdown: 점수 구성 요소
metrics: 매출, 유동인구, 전환효율, 객단가 등 상세 지표
```

## 15. 보고서에서 설명하면 좋은 핵심 문장

보고서에서는 다음처럼 설명할 수 있다.

```text
본 서비스는 단순히 카페전환효율이 높은 행정동을 추천하지 않는다.
카페전환효율, 타깃 연령대 매출비율, 선택 시간대 매출비중, 객단가를 종합해 기본 추천 점수를 계산한다.
이후 이상치 보정과 데이터 신뢰도 보정계수를 적용하여 표본이 작거나 극단값에 의해 왜곡될 수 있는 상권의 순위 영향을 낮춘다.
특히 신뢰도 60점 미만 상권은 검토 후보로 분리하여, 점수가 높더라도 안정 추천 상단에 노출되지 않도록 했다.
따라서 추천 결과는 창업 의사결정의 확정 답안이 아니라, 후보 상권을 1차 선별하고 추가 현장조사를 유도하는 데이터 기반 의사결정 보조 도구로 해석해야 한다.
```

## 16. 현재 구현의 한계

현재 구현은 MVP 기준이므로 다음 한계가 있다.

```text
임대료 데이터가 없다.
경쟁 카페 수가 없다.
프랜차이즈 비율이 없다.
폐업률이 없다.
도로/역세권 접근성 데이터가 없다.
요일별 매출 패턴이 없다.
행정동 단위라 실제 점포 입지보다 공간 해상도가 넓다.
```

따라서 추천 점수는 최종 창업 판단이 아니라 후보 상권 탐색용으로 사용해야 한다.

## 17. 향후 개선 방향

추가하면 좋은 개선 방향은 다음과 같다.

```text
1. 임대료와 권리금 데이터를 추가해 수익성 점수 계산
2. 카페 점포 수를 추가해 경쟁밀도 계산
3. 폐업률을 추가해 리스크 점수 계산
4. 지하철역/버스정류장 접근성 반영
5. 지도 기반 시각화 추가
6. 타깃 전환효율을 점수에 반영할지 실험
7. 점수 가중치를 사용자가 직접 조정하는 기능 추가
```
